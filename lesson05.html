Animatie oorlogen

SMIL, Web-Animation, CSS-Animation, javascript

Tot nog toe hebben we dingen geanimeerd met javascript. Dit is ook handig als er interactie in je SVG is.

Voor animaties, zijn er echter een aantal alternatieve systemen, SMIL door applle, Web-Animations Api door Chrome en CSS-Animation dat nooit bedoeld was om SVG mee te animeren en daarom niet alle attributes kan animeren, maar SVG is nu eenmaal te stylen met CSS en daar hoort dus ook CSS-animation bij.

Het grote probleem met al deze standaarden, is dat niemand het er (nog) over eens lijkt te kunnen worden. Chrome had SMIL op deprecated staan, maar heeft dat weer ingetrokken na veel druk uit de community die SMIL voorstaat. Web-Animations API lijkt de standaard die het meest overeenkomt met het principe van "seperation of concerns", wordt (nog) niet ondersteund door Edge etc etc.
Het is doffe elllende.


Waarom Vanilla Javascript?

Daarom ga ik in deze tutorial uit van (vanilla) Javascript. Dit heeft een aantal voordelen.
  • Javascript is zowel voor interactie als animatie geschikt
  • Javascript geeft meer controle over de animatie, je kunt stoppen op het moment dat je wilt, wijzigingen maken etc. Het is meer geschikt voor interactieve animaties, wat op het web mooi meegenomen is en voor Games onmisbaar.
  • Er zijn goede libraries als greensock en D3 beschikbaar om je te helpen met moeilijke dingen, maar ik ben geen fan van automatisch animeren van libraries. Zodra de Browser-Wars zijn opgelost, zullen de meeste animatie-libraries hun bestaansrecht verliezen.
  • Je hebt geen last van dit soort browser-strijd. Dit omdat Ecmascript (de makers van Javascript) als uitgangspunt hebben: "Don't break the web". Oude versies van javascript blijven het gewoon doen, ook al komt er functionaliteit bij.
Het enige nadeel is, dat Javascript bij echt simpele animaties wat groter uitpakt (meer Kb). Bij de meeste complexere animaties wint je bij puur javascript weer Kb terug..
